# AVIF 支持 + SW 通知丢失竞态修复 + demo 改造

## 变更原因

用户先要求给 picman 加入 AVIF 动图支持(此前发现某 CDN 资源在浏览器内容协商下会返回 AVIF 而非 WebP,picman 之前无法识别)。加入 AVIF 支持后,demo 页面重写为"直接写 `<img src>` 触发零改造接管"的形式并验证时,发现跨域 CDN 图片会永远卡在占位符阶段,深入排查后定位到一个此前从未被发现的、影响所有格式(不只是 AVIF)的严重时序竞态 bug。

## 目的

1. 让 picman 能识别并处理 AVIF 动图(含首帧提取)
2. 定位并修复"图片永远卡在占位符,三段加载不会推进到 first-frame/complete"的问题
3. 重写 demo,用真实的零改造接管场景(而非显式 API 输入框)验证 auto()、并加入 autoplay 视频 IntersectionObserver 触发时机的可视化验证

## 落地方案

### 1. AVIF 支持(`src/shared/walkers/avif.ts`)

AVIF 是 ISOBMFF/HEIF 容器,和 GIF/APNG/WebP 完全不是一个体系,不能用同样的线性扫描方式处理:

- `scanAvif()`:解析 `ftyp` box 判定 major/compatible brand 是否含 `avis`(动图序列);animated 时下钻 `moov > trak(第一个) > mdia > minf > stbl`,从 `stsd` 拿到显示尺寸与 `av1C` 编解码配置,从 `stco`/`stsz` 算出首样本字节区间(样本 1 恒为 chunk 1 起始样本,无需解析 `stsc`)
- `avifFirstFrame()`:把首样本重打包为合法的单帧静态 AVIF(`meta`/`iloc`/`iprp` 的 HEIF item 结构,不含 `moov`),重打包后自身会被 `scanAvif` 判定为 `static`,形成天然的回归验证
- `sniff.ts`/`pipeline.ts` 接入 avif 分派;因为 AVIF 首样本的字节可能在 `moov` 解析完成后才真正到达(不像 GIF/APNG/WebP 那样首帧边界发现时字节必然已在缓冲区内),`tryRecomposeFirstFrame` 针对 avif 加了显式的字节可用性边界检查
- `test/avif.test.ts` + `fixtures.ts` 里的 `makeAvif()` 程序化生成最小合法 AVIF fixture

### 2. Content-Type 不匹配 bug(`src/sw/pipeline.ts`)

`background()` 缓存首帧时,`Content-Type` 响应头错误地沿用了原始格式的 mime(如 `image/avif`/`image/webp`),但 `makeFirstFrame` 实际上无论源格式是什么,都会通过 `canvas.convertToBlob({type:"image/png"})` 光栅化成 PNG。修复为固定用 `"image/png"`。此 bug 对所有格式都存在,只是不同浏览器的内容嗅探容错程度不同,没有被更早发现。

### 3.(关键)SW 通知静默丢失竞态(`src/sw/pipeline.ts` + `src/sw/index.ts`)

**现象**:demo 里内联在 HTML 中的 `<img src="跨域CDN URL">`,SW 确实正确拦截并完整跑完了整个下载+缓存流程(Cache Storage 里 `ff`/`1` 两个阶段都写入成功),但页面的 `auto()`/`load()` 永远收不到 `first-frame`/`complete` 的 postMessage 通知,图片永远卡在占位符阶段。

**排查过程**:通过临时诊断代码逐步排除了"CORS 导致 opaque response"(实际是遗留问题但确认 crossorigin 正确设置后依然复现)、"跨域本身导致 postMessage 被 Chrome 隔离"(用不同端口的本地服务器做同源/跨域/AVIF/WebP/限速网络的多组对照实验,发现同域 100% 正常、真正的跨端口跨域也 100% 正常、连本地提供的 AVIF 内容都正常——只有"内联 HTML 里的 img 标签 + 首次访问"这个组合会复现)。最终定位:HTML parser 的 preload scanner 会独立于脚本执行顺序,在解析到 `<img>` 标签时立即发起请求;如果 SW 处理(含首帧解码)得足够快,可能在页面 `<script type="module">` 里 `await registerPicmanSW()` + `auto()` 真正完成 `message` 监听器注册之前,就已经完成下载并调用了 `notify()`。

进一步排查发现 `background()` 调用 `deps.notify(...)` 时**没有 `await`** ——`notify` 内部是异步的(`clients.matchAll().then(...)`),`postMessage` 到 Client 是跨进程操作需要真实时间;不 await 意味着 `e.waitUntil()` 延长的生命周期没有真正覆盖这段跨进程投递过程,浏览器可能在消息真正送达前就回收了 SW 实例,导致通知静默丢失(不抛异常,`clients.matchAll()` 明明能查到 client、`postMessage()` 调用本身也"成功"返回,但对方就是收不到)。

**修复**:

- `PipelineDeps.notify` 类型从 `(msg) => void` 改为 `(msg) => Promise<void>`
- `background()` 内三处 `deps.notify(...)` 调用点全部补上 `await`

### 4. `auto()` 增加 Cache Storage 对账兜底(`src/page/auto.ts`)

即便修复了上述竞态,也不该让 `auto()` 完全依赖 postMessage 这一条通路——`load()` 早就有对应的 reconciliation 机制(检查 Cache Storage 补偿错过的通知),`auto()` 的 `track()` 之前却只依赖内存态 `stageOf`,一旦消息真的丢失就永远没有恢复手段。新增 `reconcileFromCache()`:track 一个尚不知道阶段的 URL 时,按顺序 await 检查 `ff` 后 `1` 两个缓存条目,命中就补上对应 swap——顺序与 `load()` 一致,保证不会因为对账而跳过首帧直接展示完整(重)内容,避免对 LCP 不友好。

### 5. demo 页重写(`examples/index.html`)

- A 区:跨域 CDN 图片,直接写 `<img src="...">`,不经过任何显式 API,验证零改造接管
- C 区:同域 `giphy.gif` 作为对照组
- B 区:`<video autoplay poster>` 前面放一大段占位内容把它推出首屏视口,验证 `after-lcp` 策略下,视频不在视口内时会转为用 IntersectionObserver 监视,真正滚动进入视口才 restore 播放,而不是页面刚加载、用户还看不到时就抢占带宽自动播放
- 用 `MutationObserver` 观察 img/video 的 src/poster/preload 属性变化,配合相对时间戳日志,替代此前"输入框+按钮+load() API"的交互方式,更贴近真实使用场景
