# 2026-07-16 v1 动图渐进加载实现落地(Task 1~13)

## 变更原因

`docs/plans/PLAN-2026-07-15-v1-animated-progressive.md` 的实现计划已按 Task 1~13 逐条落地完成。原 0.1.0 的内存元数据管理器(`Picman` 类)已整体移除,转型为 SW 拦截动图请求的渐进加载库。

## 目的

把 REQUIREMENTS.md 中记录的 v1(动图)需求变成可运行、可测试的代码,并同步实现与需求之间的两处已知偏差。

## 落地方案

- `src/shared/`:字节工具(`bytes.ts`)、协议常量(`protocol.ts`)、公共类型(`types.ts`)、三格式 walker(`walkers/gif.ts` / `apng.ts` / `webp.ts`)、统一嗅探入口(`sniff.ts`)。
- `src/sw/`:占位生成(`placeholder.ts`)、LRU 缓存层(`cache.ts`)、状态机管线(`pipeline.ts`)、自装配组装(`index.ts`)。`src/sw.ts` / `src/sw-standalone.ts` 分别对应自装 / 托管两种部署方式。
- `src/page/`:消息总线(`messages.ts`)、显式 API(`load.ts`)、SW 注册辅助(`register.ts`)、零改造接管(`auto.ts`)。
- `src/element/`:`<pic-man>` Web Component,内部基于 `load()`。
- `examples/`:demo 页 + 限速静态服务(`serve.ts`,`node --experimental-strip-types` 运行,不参与包构建)。

## 与需求的已知偏差(按 REQUIREMENTS.md 已提前声明)

1. **CSS 背景图接管**:`auto()` 仅支持 `data-picman-bg="<url>"` 显式标记元素,不做样式表扫描(YAGNI,REQUIREMENTS.md 已注明"设计经脑暴但延后")。
2. **GIF 动图判定**:仅凭 Netscape 循环扩展不判定为动图,须至少出现两帧图像块才判 `animated`——原因是若仅凭循环扩展判定,首帧截断重组产物(单帧 + 遗留的循环扩展)会被误判回动图,导致"首帧占位"这一档本身无法降级为合法静图。此判定口径在 `src/shared/walkers/gif.ts` 内联注释中说明。
3. **`auto()` 的 `<img>` 接管范围**:v1 只接管 `src` 属性,不处理 `srcset` / `<picture><source>`——留待后续版本按需补充。

## 验证

`pnpm test`(58/58 通过)、`pnpm build`(4 ESM 入口 + `dist/picman-sw.js` 齐全)、`pnpm check`(fmt 净 + lint 0 error,99 条非阻塞 warning,集中在测试文件的 mock 类型断言,可接受)。

补充验证(同日):`examples/big.gif` 用程序化脚本生成(60 帧、724KB、结构合法但非真实画面),跑通 `examples/serve.ts --throttle 51200`——`curl` 下载 724KB 耗时约 15.6s(限速 50KB/s 符合预期)且字节与源文件完全一致,`index.html`/`picman-sw.js`/`big.gif` 三个路径均 200;另用该文件真实字节跑 `sniff()`,确认 4096 字节头部返回 `need-more`(GCT+首帧未收全),完整字节返回 `animated` 且宽高、`gifFirstFrameEnd` 均正确。真实浏览器里观察色块→静态首帧→动画三段切换(依赖 Service Worker + `<img>` 渲染)未做,因当前环境无浏览器可驱动,留给有 GUI 环境时人工过一遍。
