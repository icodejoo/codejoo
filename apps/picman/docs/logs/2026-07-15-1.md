# 2026-07-15-1:picman 重定位为 SW 渐进图片加载库

## 变更原因

现有 `@codejoo/picman`(0.1.0)只是一个内存 Map 的图片元数据管理器(add/get/remove/getAll),与实际想要的产品完全不符。本次脑暴确定了真正的产品方向,现有实现将被整体替换。

## 目的

做一个利用 Service Worker 拦截图片请求的渐进加载库:

- 小体积图片请求直接放行;
- 大体积图片先请求到关键信息(宽高 + 极简颜色),立即合成极度模糊的占位图交给页面显示;
- 后台继续下载全图,完成后通知页面切换为清晰图;
- GIF / APNG / 动画 WebP 等动图同样处理(首帧静态模糊占位 → 全图后动画从头播)。

改善大图场景的首屏体验:先见轮廓/色调,不白屏不跳变。

## 落地方案

脑暴已确认的决策(详见 `docs/REQUIREMENTS.md`):

1. **架构**:传输无关核心 + 可插拔适配器(SW 自装 / SW 托管成品 / 页面 fetch 流式),不做三份代码。
2. **占位样式**:`placeholder: 'color' | 'thumb'` 双模式参数控制——色块 vs 微缩真图模糊。
3. **页面接入三种全要**:B 显式 API(`picman.load`)为底座,A 零改造接管(`picman.auto`,含 `<img>`/srcset/CSS 背景图)与 C Web Component(`<pic-man>`)建在 B 上。
4. **判定管线**:URL include/exclude 粗筛 → Content-Length 对阈值(默认 100KB)细判;无 Content-Length 时边读边计字节,超阈值转渐进(缓冲续用),读完未超则透传。
5. **缓存**:Cache Storage + LRU(默认 200 条 / 7 天,可配)。
6. **包结构**:单包多入口——`.`(页面运行时)/`./sw`/`./element`/`./shared`,另发 `dist/picman-sw.js` 托管成品(方案二,不拆多包)。

涉及文件:`src/` 全部重写(shared/sw/page/element 分层)、`package.json` exports 多入口、`vite.config.ts` 多入口构建、`README.md`、`test/` 重写。
