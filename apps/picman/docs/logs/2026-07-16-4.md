# LCP 友好原则:真实内容一律延后到主线程空闲才应用

## 变更原因

在验证并修复了 SW 通知丢失竞态(见 `2026-07-16-3.md`)后,用户提出一个更进一步的架构原则:无论 first-frame/complete 是通过 SW 实时通知拿到的,还是通过 Cache Storage 对账(reconciliation)瞬间命中的,都不应该在页面 LCP(Largest Contentful Paint)完成之前就把真实数据源(图片/视频)替换上去。

此前 `load()`/`auto()` 的对账逻辑虽然已经保证了"先 first-frame 后 complete"的展示顺序(避免跳过首帧直接展示更重的完整内容),但没有考虑"重复访问导致的瞬间缓存命中"本身也可能抢在 LCP 之前发生——如果这张图恰好是页面的 LCP 候选元素,缓存命中越快,反而可能因为要在关键渲染路径上多解码一次(哪怕是首帧)而拖慢 LCP。

## 目的

让"真实内容替换"这个动作,始终发生在页面关键渲染路径完成之后,不管这个真实内容是通过网络下载来的、还是本地缓存瞬间给的。

## 落地方案

### 1. 复用 video facade 已有的 LCP 近似策略

`src/page/video.ts` 里 autoplay 视频的 `videoAutoplay: "after-lcp"` 策略,一直是用 `requestIdleCallback`(无该 API 时退回 `setTimeout`)来近似"LCP 已经过去"——主线程进入空闲通常意味着页面的关键渲染工作已经做完。这个思路被提取为共享模块 `src/page/idle.ts` 的 `scheduleIdle(cb, timeout = DEFAULT_IDLE_TIMEOUT)`,`video.ts` 改为从这里导入,不再各自维护一份。

### 2. `load()` 的三处"真实内容 emit"点全部延后

`src/page/load.ts`:

- SW 实时通知的 `first-frame`/`complete` 分支
- Cache Storage 对账命中的两个分支

都从"立即 `emit(...)`"改为"`scheduleIdle(() => emit(...))`"。`unsubscribe()` 等内部簿记动作仍然立即执行(避免继续接收无关消息),只有"通知调用方"这一步被推迟。下载失败的 `error` 分支不受影响,仍然立即 `rejectDone`,不能把负面信号也故意拖延。

### 3. `auto()` 新增 `applyStageWhenIdle`,统一收敛所有"应用真实阶段"的入口

`src/page/auto.ts` 新增:

```ts
function applyStageWhenIdle(url: string, stage: PicmanStage): void {
  scheduleIdle(() => {
    if (stage === "ff" && stageOf.get(url) === "1") return; // 防止阶段倒退
    stageOf.set(url, stage);
    swapAll(url, stage);
  });
}
```

`onMessage`(SW 实时通知)、`reconcileFromCache`(缓存对账)、`onVisible`(页面重新可见时的缓存检查)三处原本各自直接 `stageOf.set` + `swapAll` 的地方,统一改为调用这个函数。

**防止阶段倒退**是这里的关键细节:因为 `first-frame` 和 `complete` 各自独立调度了一次 `scheduleIdle`,如果两者几乎同时到达,不能保证对应的 idle 回调一定按到达顺序执行——如果 `complete` 的回调先跑完(页面已经在显示完整内容),之后 `first-frame` 的回调才姗姗来迟,绝不能把已经完整显示的内容降级回首帧占位。回调执行时重新检查 `stageOf` 的当前值来把关。

`track()` 函数里"阶段已知则立即 `applyUrl`"这条错过通知补偿路径不需要改代码——因为 `stageOf` 本身现在只会在 idle 回调里才被设置,所以这条路径天然不可能在 LCP 之前触发。

### 4. 测试基础设施调整

`test/load.test.ts`、`test/auto.test.ts`、`test/element.test.ts` 在 `beforeAll` 里把全局 `requestIdleCallback` 替换成 `(cb) => setTimeout(cb, 0)`,这样 `scheduleIdle` 默认的 2000ms 超时不会拖慢测试——一轮宏任务(`await new Promise(r => setTimeout(r, 0))`)即可等到"空闲回调"跑完。新增 `test/idle.test.ts` 覆盖 `scheduleIdle` 自身(有/无 `requestIdleCallback` 两种路径、超时透传)。

全量测试(87 个用例)与 `pnpm run check` 均通过。
