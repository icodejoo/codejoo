# LCP 与用户体验的折中:编码感知双路径缩略图 + 真实 LCP 信号 + offViewport

## 变更原因

静态渐进(见 `2026-07-16-5.md`)首版用"截断字节直接给 `<img>` 宽容解码"做缩略图,用户实测反馈:baseline JPEG / 非隔行 PNG 只显示顶部一小条,体验太差。随后确立了新流程(占位 → 下载完出缩略图 → 进视口给高清 → 离开视口按参数回退),并追问该流程是否阻塞 LCP、以及 LCP 指标与"用户尽早看到真实内容"之间有无折中。最终确定同时实施两个折中方案:①编码感知分流,②真实 LCP 信号感知调度。

## 目的

1. 缩略图体验:永远不给"只有顶部一条"的半截图;能全图覆盖时尽量早给
2. LCP:保持关键渲染路径上只有色块占位,真实内容的切换精确地发生在 LCP 之后
3. 视口管理:高清进入视口才给;离开视口后的行为参数化

## 落地方案

### 1. 编码感知双路径缩略图(方案①)

`sniff.staticDisplayable` 语义收紧为"已到字节能否解出**全图覆盖**的预览":

- **渐进式 JPEG**(SOF2):首个 scan 收完(`scanJpeg.firstScanEnd`)→ 全图模糊 → 触发,把已到原始前缀字节缓存为 `ff`——远早于下载完成(通常 10-20% 字节处)
- **隔行 PNG**(Adam7,`scanPng` 新增 `interlaced` 检测 IHDR 第 13 字节):IDAT 累计跨过 4KB → 全图马赛克 → 同上
- **baseline JPEG / 非隔行 PNG**:永不触发早期信号;`backgroundStatic` 下载完成后,用 `makeFirstFrame`(createImageBitmap + OffscreenCanvas,SW worker 线程,不占主线程)光栅化一张降采样缩略图(≤512px PNG)缓存为 `ff`

两条路径产出的都是 `ff` 阶段,页面端零协议改动。README 新增一节建议业务方把大图转渐进式编码(构建期一行配置)以获得最佳体验,hero 图用 `exclude` 排除。

### 2. 真实 LCP 信号感知调度(方案②,`src/page/idle.ts`)

`scheduleIdle` 从"纯 requestIdleCallback 近似"升级为双信号:模块加载时用 `PerformanceObserver` 监听 `largest-contentful-paint`(`buffered: true` 补早于脚本的绘制),工作在"主线程空闲 **且** 已有 LCP 条目绘制"两者较晚者执行;idle 先到而 LCP 未至时以 100ms 轮询等待,超 1 秒(`LCP_EXTRA_WAIT`)兜底执行,保证不产生 LCP 条目的页面不卡死。不支持该 entry 类型的环境用 `PerformanceObserver.supportedEntryTypes` 检测(部分环境 observe() 不抛错但永不产出条目,不能只靠 try/catch),退回纯 idle 门控与此前行为一致。

### 3. `offViewport` 离开视口回退(`src/page/auto.ts` + `src/page/types.ts`)

`auto({ offViewport })` 三档:

- `'keep'`(默认):进入视口给高清后一次性完成,不再观察——无回切、无闪烁
- `'thumbnail'`:高清元素保持被 IntersectionObserver 观察,离开视口回退到 `ff` 阶段 URL(Cache 命中零网络),浏览器得以释放完整大图的解码内存;再次进入恢复高清
- `'placeholder'`:离开视口回退到页面端按元素尺寸生成的 SVG 色块 data URI(`svgColorBlock`/`svgDataUri` 来自 shared,零网络)

配套把回声判定(`isOwnEcho`)从"按 URL 的最新阶段"升级为"按元素的 `elExpected`"——引入回退后元素 src 可以合法地不同于其 URL 的最新阶段,原逻辑会把我们自己的回退写入误判为外部变更,触发 track → 立即切回高清的反馈回路。新增 `elUrl` WeakMap 记录元素↔规范化 URL 映射,供 IO 回调恢复高清用。

## 验证

- 全量 108 个测试通过(新增:jpeg/png 编码分流信号、pipeline 双路径产物断言、idle 的 LCP 双信号/兜底/回退、auto 的 offViewport 三档回退与反馈回路防护)
- Chrome 实测(400KB/s 限速):baseline JPEG(768KB)`ff` 出现在 +2055ms(下载完成时刻),Cache 中 ff 条目为 `image/png`、PNG 魔数、438KB 光栅化缩略图——双路径正确;`PerformanceObserver.supportedEntryTypes` 含 `largest-contentful-paint`,真实信号路径生效
- 排查插曲:用 `fetch()` 验证 `ff` 内容时拿到完整 JPEG,一度误判为 bug——实际是 `fetch()` 的 `destination` 为空字符串,`shouldIntercept` 不拦截,请求直达静态服务器(忽略 query 参数);Cache Storage 内容才是管线真实产物,验证 SW 缓存应直接查 `caches` 而非发 `fetch()`
