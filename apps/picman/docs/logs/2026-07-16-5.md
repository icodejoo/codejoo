# 静态大图(PNG/JPEG)渐进加载 + 高清视口门控

## 变更原因

用户提出:希望静态大图也能渐进显示——比如一张 100KB 的 PNG,拦截住请求,一旦已到达的字节足以显示出内容(如 20KB 时),就先把这个低质量预览(缩略图,模糊也可以)渲染给用户,直到元素真正进入可见区,才显示高清完整图。并明确澄清:触发缩略图的门槛**不是固定字节数**,而是动态的——"什么时候可以显示了"就什么时候切。

## 目的

1. 把 picman 的渐进加载能力从动图(GIF/APNG/WebP/AVIF)扩展到静态大图(PNG/JPEG)
2. 缩略图触发时机采用逐图动态的结构性信号,而非固定字节阈值
3. 高清完整图按元素粒度做视口门控:视口外的元素停留在预览态,真正滚入可见区才换高清

## 前置调研结论(浏览器实测)

- `createImageBitmap` 对截断的 PNG/JPEG 字节零容忍(`InvalidStateError`)——SW 端无法用现有 `makeFirstFrame` 路径从部分字节渲染缩略图
- **`<img>` 元素的解码器宽容得多**:截断到 15% 的 baseline JPEG、截断到 20% 的 PNG 都能解码显示(已到部分可见),`naturalWidth/Height` 返回完整尺寸
- 因此方案定为:SW 把已到达的**原始前缀字节**直接缓存为 `ff` 阶段(正确 MIME),由页面 `<img>` 的宽容解码器渲染——不做 SW 端重组

## 落地方案

### 1. 动态"可显示"信号(`sniff.staticDisplayable`)

不同编码方式的最早可显示点不同,按结构判定:

- **渐进式 JPEG**(SOF2):第一个 scan 收完(熵数据后出现下一个真实 marker)→ 全图覆盖的模糊图像可解,这是最优情形
- **baseline JPEG**:越过 SOS、熵编码数据攒到 `MIN_PIXEL_BYTES`(4096)→ 顶部切片可解
- **PNG**:IDAT 负载累计字节(含未收全的尾部 chunk 的已到部分)≥ `MIN_PIXEL_BYTES` → 顶部若干行可解

对应改动:

- 新增 `src/shared/walkers/jpeg.ts`:`scanJpeg` 解析 SOF 尺寸(区分 SOF0/SOF2)、`scanDataStart`(SOS 后熵数据起点)、`firstScanEnd`(渐进式首 scan 结束点,穿透 0xFF00 填充与 RSTn)
- `scanPng` 增加 `idatBytes` 计数
- `sniff` 统一输出 `staticDisplayable` 布尔信号,并新增 `jpeg` 格式分派(`FFD8` 魔数)

### 2. SW 端静态渐进流程(`pipeline.ts` 的 `backgroundStatic`)

`mainFlow` 里 `status === "static"` 分支不再一律透传:当格式为 jpeg/png、尺寸已知、`staticProgressive` 开启(默认 true)时,立即返回 SVG 色块占位,`waitUntil(backgroundStatic(...))` 后台继续:

- 每读一块就重新 `sniff`,`staticDisplayable` 首次为 true 时,把**当时已累积的原始前缀字节**缓存为 `ff` 阶段(Content-Type 用真实 MIME)并发 `first-frame` 通知——这就是"动态门槛":头部小、渐进编码的图可能几 KB 就触发,EXIF/ICC 巨大的图要等更久
- 下载完成后缓存 `1` 阶段 + `complete` 通知

未识别格式仍原样透传;`staticProgressive: false` 时恢复旧行为(透传)。默认 include 规则加入 `jpe?g`。

### 3. 页面端高清视口门控(`auto.ts`)

`swapAll`/`track` 的阶段应用统一收敛到 `applyStageToElement`:`'1'`(高清)阶段按元素粒度检查——在视口内(或 IntersectionObserver 不可用)立即应用;视口外则挂到共享 IntersectionObserver 上,真正滚入可见区才换 src。`ff`(缩略图)不门控,照常在 idle 后应用。`stop()` 断开该观察器。

复用消息协议的 `first-frame`/`complete` 与 `ff`/`1` 阶段参数,零协议改动;LCP 门控(`applyStageWhenIdle`)原样生效。

## 验证

- 单元测试:`test/jpeg.test.ts`(SOF/SOS/首 scan/动态信号)、pipeline 静态渐进三段流程、`staticProgressive: false` 关闭行为、动图路径不受影响、auto() 视口门控(视口外扣住 complete、进入视口切换)——全量 101 个用例通过
- 端到端(Chrome 实测,`examples/big.png` 为程序化生成的 412KB 真实可解码 PNG):
  - 视口内:初始 → `ff`(+23ms)→ `1`(+39ms)三段完整触发
  - `ff` 阶段响应可被 `<img>` 正常解码
  - 视口外元素:`complete` 到达后 src 停留在 `ff`,`scrollIntoView` 后立即切换到 `1`
  - 注:本地不限速时下载快到没有中间态,`ff` 可能等于完整字节;慢网络下才会出现真正的截断缩略图,这正是该功能的目标场景
