02

拆解两处关键决策

已修复的竞态

background() 调用 deps.notify(...) 曾经没有 await。notify 内部要做 clients.matchAll() 和跨进程 postMessage,这需要真实时间——不等待意味着 waitUntil() 延长的生命周期没有真正覆盖投递过程,浏览器可能在消息送达前就回收了 SW 实例,通知因此静默丢失,图片永远卡在占位符阶段。

触发条件很刁钻:HTML parser 的 preload scanner 独立于脚本执行时序发起请求,如果下载/通知发生得足够快,可能抢在页面 auto() 完成 message 监听器注册之前——三处 notify() 调用现已全部补上 await。

LCP 门控原则

无论 first-frame / complete 是通过 SW 实时通知拿到的,还是 Cache Storage 瞬间命中的,一律不在页面主线程空闲(近似 LCP 完成)之前生效——重复访问的瞬间缓存命中,不该抢在关键渲染路径前面替换成解码更重的内容。
